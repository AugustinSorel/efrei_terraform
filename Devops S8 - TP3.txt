1. Présentation du TP
1.1 Contexte et objectifs
Ce TP fait suite au cours Ansible. Vous allez maintenant ajouter la couche infrastructure : plutôt
que de supposer qu'un serveur existe et qu'Ansible s'en occupe, vous allez le créer vous-même
dans AWS avec Terraform.

1.2 Ce que vous allez construire
L'objectif final est une infrastructure cloud fonctionnelle comprenant :
•​

Un VPC avec subnets public et privé dans AWS

•​

Une instance EC2 (serveur web) avec Security Group configuré

•​

Une paire de clés SSH gérée par Terraform

•​

Un bucket S3 (stockage fichiers statiques ou remote state)

•​

En bonus : une base de données RDS et un Load Balancer

⚠️ Attention

L'ensemble de votre code Terraform sera versionné dans un dépôt Git. Terraform destroy sera
lancé à la fin de chaque séance pour ne pas consommer les crédits AWS inutilement.

3. Mise en place et infrastructure de base
3.1 Mise en place de l'environnement
Cette partie se déroule en deux phases distinctes :
●​ Phase 1 - LocalStack : vous écrivez et testez tout votre code Terraform localement, sans
compte AWS, sans coût, sans risque.
●​ Phase 2 - AWS réel : une fois le code validé sur LocalStack, vous le déployez sur AWS
avec vos crédits ($100 offerts à l'inscription).

💡 Conseil

Faites toujours valider votre code sur LocalStack avant de l'appliquer sur AWS réel. Cela évite de
consommer des crédits sur du code qui ne fonctionne pas encore.

Phase 1 - Installer et démarrer LocalStack
LocalStack est un émulateur AWS qui tourne dans Docker sur votre machine. Il simule les
services AWS (EC2, S3, VPC, RDS...) sans aucun compte.
Prérequis : Docker doit être installé et démarré sur votre machine.
Installer LocalStack ainsi que tflocal (terraform-local)
# Démarrer LocalStack (en arrière-plan)
localstack start -d
# Vérifier que les services sont prêts
localstack status services
curl http://localhost:4566/_localstack/health

ℹ️ Note

LocalStack Community Edition (gratuite) supporte : EC2, S3, VPC, IAM, SQS, DynamoDB,
Route53, ACM, et bien d'autres. Pour RDS, l'édition Pro est requise - utilisez directement AWS
réel pour la BDD.

Phase 1 - Utiliser tflocal à la place de terraform
tflocal est un wrapper de la commande terraform qui redirige automatiquement toutes les
requêtes vers LocalStack (http://localhost:4566) sans modifier votre code HCL.
# Remplacer terraform par tflocal pour la phase LocalStack
tflocal init
tflocal plan
tflocal apply
tflocal destroy

Pour LocalStack, Terraform a besoin de fausses credentials (n'importe quelle valeur) :
export AWS_ACCESS_KEY_ID=test
export AWS_SECRET_ACCESS_KEY=test
export AWS_DEFAULT_REGION=us-east-1

⚠️ Attention

Certaines ressources ne se comportent pas exactement comme sur AWS réel dans LocalStack.

Phase 2 - Créer votre compte AWS et configurer les credentials
Une fois votre code validé sur LocalStack, vous allez le déployer sur AWS réel.
1.​ Créez un compte AWS sur aws.amazon.com (carte bancaire requise pour vérification,
mais $100 de crédits offerts à l'activation)
2.​ Connectez-vous à la console AWS → Informations d'identification de sécurité → Créer
une clé d’accès
3.​ Configurez AWS CLI avec vos clés :
aws configure
AWS Access Key ID
: AKIA...
AWS Secret Access Key : ...
Default region name
: eu-west-3
Default output format : json

Vérifiez que la connexion fonctionne :
aws sts get-caller-identity

Phase 2 - Passer de tflocal à terraform
Aucune modification de votre code HCL n'est nécessaire - remplacez simplement tflocal par
terraform dans vos commandes :
# Phase 2 : commandes standard Terraform
terraform init
terraform plan
terraform apply

🚫 Important

Les crédits AWS ($100) sont partagés sur toute la durée du cours. Lancez impérativement
terraform destroy à la fin de chaque séance. Une instance EC2 t3.micro oubliée coûte ~$0.01/h
soit ~$7/mois - multipliez par toute la classe.

3.3 Configurer le provider AWS
Remplissez versions.tf avec la configuration Terraform et le provider requis :
# versions.tf
terraform {
required_version = ">= 1.6"
required_providers {
aws = {
source = "hashicorp/aws"
version = "~> 5.0"
}
}
}

Puis le fichier provider.tf :
# provider.tf
provider "aws" {
region = var.aws_region
default_tags {
tags = {
Project
= var.project_name
Environment = var.environment
ManagedBy
= "Terraform"
Owner
= var.owner
}
}
}

Le bloc default_tags est une bonne pratique AWS : toutes vos ressources hériteront
automatiquement de ces tags, sans avoir à les répéter sur chaque ressource.
Initialiser le projet :
terraform init

Terraform va télécharger le provider hashicorp/aws (~100 Mo). Vérifiez qu'il n'y a pas d'erreur. Le
dossier .terraform/ est créé - il est dans le .gitignore.

4 Travaux bonus
Bonus A - Base de données RDS
Ajoutez un subnet privé et une instance RDS PostgreSQL. La BDD ne doit PAS avoir d'IP
publique et ne doit être accessible que depuis le Security Group de l'EC2.
Ressources à créer :
•​

aws_subnet (privé, map_public_ip_on_launch = false)

•​

aws_db_subnet_group (requis par RDS)

•​

aws_security_group pour la BDD (port 5432 depuis le SG EC2 uniquement)

•​

aws_db_instance (db.t3.micro, postgres, skip_final_snapshot = true)

Ajoutez un output db_endpoint et passez-le à Ansible via le template d'inventaire.

Bonus B - Application Load Balancer
Ajoutez un ALB devant l'EC2. L'ALB sera le point d'entrée public, l'EC2 n'aura plus besoin d'être
directement accessible sur le port 80.
Ressources à créer :
•​

Deuxième subnet public dans une autre AZ (obligatoire pour l'ALB)

•​

aws_lb (type = application)

•​

aws_lb_target_group (port 80, protocol HTTP)

•​

aws_lb_listener (port 80 → forward vers le target group)

•​

aws_lb_target_group_attachment (associe l'EC2 au target group)

Bonus C - Terraform Workspaces (Multi-environnements)
Les workspaces permettent de gérer plusieurs environnements avec le même code. Créez un
workspace dev et un workspace prod avec des tailles d'instances différentes.
terraform workspace new dev
terraform workspace new prod
terraform workspace select dev

Adaptez les variables selon le workspace dans variables.tf :
locals {
instance_type = {
dev = "t3.micro"
prod = "t3.small"
}
}
# Utiliser : local.instance_type[terraform.workspace]

Bonus D - tflint
tflint est un linter Terraform qui détecte les erreurs de configuration AWS (types d'instances
invalides, AMIs dépréciées, règles de sécurité...).
